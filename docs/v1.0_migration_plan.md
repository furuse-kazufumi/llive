# v1.0.0 Migration Plan — FullSense Brand Unification

> Status: **Draft** (2026-05-16). 実施は v1.0.0 リリース時 (時期未定).
> 本書は v0.6.0 〜 v1.0.0 の間に確定すべき決定事項と移行手順を整理する RFC.

## 背景

v0.6.0 で「FullSense」を umbrella ブランドとして導入したが、PyPI パッケージ
名は依然として `llmesh-llive` / `llmesh-llove` のままで、ブランドと不一致
が残る。v1.0.0 で **FullSense brand と PyPI 命名を整合させる** ことを目的
とする RFC。

## 命名の現状と目標

| Layer        | v0.6.0 (現在)     | v1.0.0 (目標)              |
|--------------|-------------------|-----------------------------|
| Umbrella     | (未公開)          | `fullsense` (meta package)  |
| Hub          | `llmesh`          | `fullsense-llmesh`          |
| Memory       | `llmesh-llive`    | `fullsense-llive`           |
| TUI          | `llmesh-llove`    | `fullsense-llove`           |
| Suite        | `llmesh-suite`    | `fullsense-suite`           |
| Import path  | `import llive`    | `import llive` (不変)       |
| GitHub       | `furuse-kazufumi/llive` 他 | 不変 (個人 repo 維持) |

**import path は不変** — ユーザコードに影響を与えない。PyPI 配布名のみ rename.

## 移行手順 (v1.0.0 cut 直前 → 直後)

### Phase 1: 並行公開 (1 minor バージョン前から)

  1. `fullsense-llive` を **新規 PyPI 名で予約公開** (v0.7.0 / v0.8.0 で予約 push)
  2. 旧 `llmesh-llive` の README に「次の major で fullsense-llive に rename」と告知
  3. CHANGELOG に DEPRECATED section 追加

### Phase 2: v1.0.0 cut

  1. `fullsense-llive` v1.0.0 を **primary** として公開
  2. 旧 `llmesh-llive` v1.0.0 を **alias / shim** として公開 (依存関係に `fullsense-llive>=1.0.0` を 1 行)
  3. README / docs / pyproject.toml の package name を全て `fullsense-*` に書き換え
  4. classifiers, keywords も同期

### Phase 3: v1.x maintenance

  1. 旧 `llmesh-*` は v1.x で**新規機能追加なし**, fullsense-* のみメンテ
  2. v2.0.0 で旧 `llmesh-*` を **yank** または「last release was 1.0.0 」状態で停止
  3. README に「Renamed to fullsense-*」のみ残す

### Phase 4: 商標登録完了後

  1. `™` → `®` 表記更新 (jurisdiction 単位)
  2. NOTICE / TRADEMARK.md の Status 列を更新

## ユーザ向け移行手順

```bash
# 旧
pip uninstall llmesh-llive
# 新
pip install fullsense-llive
# import は不変
python -c "import llive; print(llive.__version__)"
```

`requirements.txt` 等は alias 経由でも動く (v1.x の間)。

## 影響範囲

- **コード変更**: import path 不変 → ユーザコード変更ゼロ
- **CI/CD 変更**: `pip install llmesh-llive` → `pip install fullsense-llive` の 1 行
- **ドキュメント**: GitHub README / Mintlify / PyPI 説明文 / 各種記事 (linkedin / qiita) の言及置換
- **Search index**: `llmesh-llive` での検索流入は v1.x 期間中も alias で受け止め

## v0.7.0 〜 v0.9.x の中間バージョンで先行できること

  - [ ] `fullsense-llive` PyPI 名予約 (空 release で確保)
  - [ ] `import llive` で `fullsense.llive as llive` の alias namespace package を試験的に追加
  - [ ] README に rename 予告を載せる
  - [ ] CHANGELOG に DEPRECATED 警告を出す
  - [ ] `pyproject.toml` に `[project.entry-points."fullsense.products"]` で metadata 公開

## 商標との関係

PyPI rename はユーザの **移行コスト** を伴う一方、商標登録は **トリガなし**
で進められる。商標出願は PyPI rename を待たずに先行可能 (`docs/legal/trademark/`).

## リスクと緩和

| リスク                                   | 緩和                                       |
|------------------------------------------|--------------------------------------------|
| ユーザの混乱 / 既存プロジェクトの破綻    | alias リリース + 移行ガイド + 1 minor 告知期間 |
| PyPI 名の squatter 占拠                  | 早期予約 (空 release で確保)               |
| `llmesh-*` 検索流入の喪失                | 旧名 README に「Renamed to ...」維持       |
| GitHub repo 名変更が必要かの判断         | 個人 repo は名前変更不要、cross-link で運用  |

## 決定が必要な項目

  - [ ] `fullsense` meta-package の中身: 単なる re-export か、CLI 統合か
  - [ ] alias 期間の長さ: 1 minor / 1 major / 永続
  - [ ] import path も将来的に `import fullsense.llive` に切替えるか
  - [ ] PyPI 名 yank のタイミング (v2.0 / v3.0)
  - [ ] 商用 license 名は `LICENSE-COMMERCIAL` のままで OK か (例: `FULLSENSE-COMMERCIAL-LICENSE`)
